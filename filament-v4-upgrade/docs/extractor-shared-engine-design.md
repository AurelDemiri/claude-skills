# Design: update the Filament-v4 upgrade extractor to emit the shared-engine shape

_Authored in PUBMUS, 2026-06-25. Companion to `synthesis-recipe.md`. Describes the change to the
VLWPLA `chore/filament-v4-upgrade-extractor` pipeline; the extractor branch is updated separately._

## Problem

The VLWPLA extractor mines a real, completed Filament v3->v4 upgrade (448 non-merge commits ->
141 deterministic candidates -> **88 knowledge cards** under
`docs/filament-v4-upgrade-analysis/extracted/*.md`) and then SYNTHESISES deliverables. Today its
agent-driven synthesis step emits two files: `filament-v4-upgrade-guide.md` (the ~78 KB catalog, keep)
and `filament-v4-upgrade.workflow.js` (the **thin serial driver** — six `await agent()` calls, no
schemas, no fan-out, no in-code gating, no self-heal).

The thin driver was superseded by the **shared-engine shape**, which already exists fully built and
tested in PUBMUS at `.claude/workflows/` (`_src/engine.mjs` ~340 lines, two `*.template.mjs`,
`_assemble.mjs`, `_wfcheck.mjs`, `_src/__tests__/`, and two generated workflows). This design makes the
extractor's synthesis step **emit that shape** instead of the thin driver, without re-deriving the
hand-tuned engineering from scratch each run.

## Decisions (locked)

1. **Fixture + knowledge slots.** The engine, templates, build tooling, and tests are a **checked-in
   scaffold** vendored from the PUBMUS golden — NOT regenerated by an LLM each run. Synthesis fills only
   the card-derived content.
2. **Guide regenerated, slots authored + validated.** Synthesis fully regenerates
   `filament-v4-upgrade-guide.md` from the 88 cards each run. The in-engine constants
   (`KNOWLEDGE()` index, `DIMENSIONS`, `FATAL_PATTERN_IDS`) stay AUTHORED in the scaffold; a **coverage
   critic** validates them against the cards and fails the gate (naming the gap) on any uncovered class,
   rather than auto-editing them.
3. **Genericized vendored defaults.** The extractor's `engine.mjs` copy ships Statik-generic
   `buildConfig` defaults overridable via `args`; the PUBMUS golden keeps its real `pubmus` defaults.
4. **Spec location.** This doc lives beside `synthesis-recipe.md` in `docs/filament-v4-upgrade/`.

## Architecture

### Pipeline boundary is unchanged

The deterministic Python harness stays exactly as-is: `_assemble.py`, `_finalize.py`/`_finalize2.py`/
`_finalize3.py`, `_fold_audit.py`, `_remediate_prep.py`, `commit-log.jsonl`, `commit-triage.jsonl`,
`_final_stats.json`. It mines commits and reconciles the 88 cards. **Only the agent-driven synthesis
step changes.** Its responsibilities become exactly three:

1. Regenerate `filament-v4-upgrade-guide.md` from the 88 cards (as today).
2. Run the **coverage critic** against the vendored engine slots.
3. Run the **build/validation gate** and only report success when green.

Its old responsibility — authoring `filament-v4-upgrade.workflow.js` — is **removed**.

### Emitted bundle layout

The extractor branch carries the golden shape as a checked-in fixture at the consumer layout, so a
target Statik repo upgrades by copying `.claude/workflows/**`:

```
.claude/workflows/
  _src/engine.mjs                         VENDORED fixture (structure + authored slots)
  _src/filament-v4-upgrade.template.mjs   VENDORED  (meta name: filament-v4-upgrade-run)
  _src/filament-v4-polish.template.mjs    VENDORED  (meta name: filament-v4-polish)
  _src/__tests__/assemble.test.mjs        VENDORED
  _src/__tests__/engine.test.mjs          VENDORED
  _assemble.mjs                           VENDORED  (--check is the drift gate)
  _wfcheck.mjs                            VENDORED  (AsyncFunction syntax check)
  filament-v4-upgrade-guide.md            REGENERATED from 88 cards each run
  filament-v4-upgrade.mjs                 GENERATED by _assemble.mjs (committed)
  filament-v4-polish.mjs                  GENERATED by _assemble.mjs (committed)
```

`docs/filament-v4-upgrade-analysis/` additionally keeps a synthesis-staging copy of the regenerated
`filament-v4-upgrade-guide.md` (as today) and gains a ported `synthesis-recipe.md`.

### Scaffold vs. regenerated content

| Artifact | Origin | Regenerated each run? |
|---|---|---|
| `engine.mjs` structure (buildConfig/RULES/schemas/helpers/stages) | vendored from golden | No |
| `engine.mjs` `KNOWLEDGE()` index lines | authored in scaffold | No (validated) |
| `engine.mjs` `DIMENSIONS` (focus prose + group) | authored in scaffold | No (validated) |
| `engine.mjs` `FATAL_PATTERN_IDS` | authored in scaffold | No (validated) |
| templates, `_assemble.mjs`, `_wfcheck.mjs`, `__tests__/` | vendored from golden | No |
| generated `filament-v4-{upgrade,polish}.mjs` | `_assemble.mjs` output | Yes (deterministic build) |
| `filament-v4-upgrade-guide.md` | 88 cards | **Yes (LLM synthesis)** |

## The slot-marker contract

The canonical `engine.mjs` (PUBMUS golden) gains three pairs of **behavior-neutral comment markers**
that bound the card-derived constants, mirroring the existing `// <engine:start>` / `// <engine:end>`
and `// <engine:inline>` convention:

- `// <knowledge:start>` ... `// <knowledge:end>` around the body of `KNOWLEDGE(cfg)`.
- `// <dimensions:start>` ... `// <dimensions:end>` around the `DIMENSIONS` array entries.
- `// <fatal:start>` ... `// <fatal:end>` around the `FATAL_PATTERN_IDS` array entries.

These are pure comments (no runtime effect; the engine still inlines correctly). They exist so the
coverage critic has stable anchors and so a future maintainer knows precisely which regions are the
card-derived slots. The PUBMUS golden is the source of truth; the extractor vendors a copy whose ONLY
delta is the genericized `buildConfig` defaults (below) plus a one-line provenance comment naming the
golden source commit.

## The coverage critic (new synthesis sub-step)

Replaces today's vague "completeness critic." It is a checker the synthesis agent runs after the guide
is regenerated.

**Inputs:** the 88 `extracted/*.md` cards, the vendored `engine.mjs` slot regions, the regenerated
`filament-v4-upgrade-guide.md`.

**For every breaking-change CLASS a card describes, assert:**
1. a `KNOWLEDGE()` index line covers it,
2. at least one `DIMENSIONS[].focus` mentions it,
3. the guide has a matching pattern section,
4. if the card is fatal/data-loss class: a `FATAL_PATTERN_IDS` id **and** a guide pattern id exist for
   it. The five current fatal ids are `namespace-move`, `form-infolist-signature`,
   `static-property-types`, `hidden-disabled-dehydrate`, `richeditor-json-state`.

**Failure behavior:** a miss does NOT auto-edit the engine. It **fails the synthesis gate** and emits a
structured report naming the exact gap — which card hash, which class, which slot is missing — so a
human authors the slot line (and, if needed, a guide pattern + fatal id). This preserves the hand-tuned
prose and makes the 88 cards the coverage oracle, not a code generator.

**Card-to-slot mapping aid:** cards already tagged with a `phase:`/category map to a
`DIMENSIONS[].group` (`breaking` | `tailwind` | `packages` | `verify`); the critic uses that tag to
pick which dimension's `focus` it expects the class to appear in.

**Implementation split (deterministic + agent).** The mechanical, reproducible invariants run as a
deterministic Python precheck (`_coverage_critic.py` in the analysis dir): markers intact and slots
parse, `FATAL_PATTERN_IDS` non-empty, and every fatal id has a matching guide pattern id. The fuzzy
semantic mapping — "does some `KNOWLEDGE()` line / `DIMENSIONS[].focus` / guide section actually cover
this card's class" — is the synthesis agent's judgment, reading the precheck output plus the cards and
slots. Both must pass for the gate to be green; either one's failure is reported as a named gap.

## The build/validation gate

After the guide is regenerated and the coverage critic passes, synthesis runs (and must pass) the
golden's own tooling, from the repo root:

1. `node .claude/workflows/_assemble.mjs` — regenerate the two `.mjs` from engine + templates.
2. `node .claude/workflows/_assemble.mjs --check` — **drift gate**: emitted == committed.
3. `node .claude/workflows/_wfcheck.mjs .claude/workflows/filament-v4-upgrade.mjs .claude/workflows/filament-v4-polish.mjs`
   — AsyncFunction syntax check (these scripts use top-level `await`/`return`, so `node --check` is
   invalid; `_wfcheck.mjs` is the correct equivalent).
4. `node --test .claude/workflows/_src/__tests__/*.mjs` — engine + assemble unit tests.

Only a fully green gate lets the synthesis run report success. This closes the "is the emitted bundle
actually assemblable and syntactically valid" gap the thin driver never had.

## buildConfig genericization (vendored copy only)

The PUBMUS golden `buildConfig` defaults stay real (`projectName: 'pubmus'`, the real
`AdminPanelProvider` and theme paths). The extractor's vendored copy makes them Statik-generic and
overridable via the workflow's `args`, e.g.:

- `projectName: a.projectName ?? 'app'`
- `panelProviders: a.panelProviders ?? ['app/Providers/Filament/AdminPanelProvider.php']`
- `themeCss`, `frontendCss`, `filamentRoot`, `vendoredOverrides`, `testDir`, `pintScope`, `packagePins`
  similarly defaulted-then-overridable.

The `gate.*` commands keep the `ddev`-prefixed Statik defaults. A consumer who passes `args` gets their
repo's shape; a consumer who passes nothing gets sane Statik conventions rather than literal `pubmus`
paths. This is the one dimension the comparison doc flagged the thin driver actually won
(reusability/parameterization), now satisfied by the engine's existing `buildConfig(args)` override
path.

## Concrete change-list

### VLWPLA `chore/filament-v4-upgrade-extractor`

- **Add** `.claude/workflows/_src/engine.mjs` (vendored from golden + genericized defaults + provenance
  comment), `_src/filament-v4-upgrade.template.mjs`, `_src/filament-v4-polish.template.mjs`,
  `_src/__tests__/assemble.test.mjs`, `_src/__tests__/engine.test.mjs`, `_assemble.mjs`, `_wfcheck.mjs`,
  and the generated `filament-v4-upgrade.mjs` + `filament-v4-polish.mjs`.
- **Delete** `.claude/workflows/filament-v4-upgrade.js` (thin driver).
- **Delete** `docs/filament-v4-upgrade-analysis/filament-v4-upgrade.workflow.js` (synthesis-staging copy
  of the thin driver).
- **Keep + regenerate** `filament-v4-upgrade-guide.md` in both `.claude/workflows/` and
  `docs/filament-v4-upgrade-analysis/` (unchanged synthesis output).
- **Port** `synthesis-recipe.md` into `docs/filament-v4-upgrade-analysis/` so the synthesis step's
  instructions are self-contained in the extractor.
- **Add** the coverage-critic instructions to the synthesis recipe + `docs/filament-v4-upgrade-analysis/_coverage_critic.py`
  (the deterministic precheck for marker/fatal/guide-id invariants; the synthesis agent handles the fuzzy
  card-class -> slot/guide semantic match).
- **Edit** `_finalize.py` (the deliverable-listing lines ~105-106) and the generated `run-summary.md`
  prose: drop "Thin workflow: `filament-v4-upgrade.workflow.js`"; describe the shared-engine bundle
  (engine + templates + `_assemble.mjs` + generated workflows + guide) and the build/validation gate.

### PUBMUS (this repo, small)

- **Edit** `.claude/workflows/_src/engine.mjs`: add the three slot-marker pairs (behavior-neutral
  comments) around `KNOWLEDGE()`, `DIMENSIONS`, `FATAL_PATTERN_IDS`.
- **Re-run** `node .claude/workflows/_assemble.mjs` and confirm `--check`, `_wfcheck.mjs`, and
  `node --test _src/__tests__/*.mjs` stay green (the markers must not change generated output beyond the
  inlined comments).
- **Update** `synthesis-recipe.md` to document the slot-marker contract and the coverage critic, so the
  recipe is the precise spec the extractor ports.

## Non-goals

- No change to the Python mining/triage harness or the 88 cards themselves.
- No change to app code in either repo.
- No live dependency/sync mechanism between PUBMUS and VLWPLA: the vendoring is a point-in-time copy with
  a provenance comment, not an automated import.
- No push / no PR is performed by the extractor or its workflows.

## Acceptance criteria

1. The extractor branch contains the full `.claude/workflows/**` shared-engine bundle and no
   `filament-v4-upgrade.workflow.js` / `filament-v4-upgrade.js` thin driver.
2. `node .claude/workflows/_assemble.mjs --check` is green on the extractor branch (generated == src).
3. `node .claude/workflows/_wfcheck.mjs` passes for both generated workflows.
4. `node --test .claude/workflows/_src/__tests__/*.mjs` passes on the extractor branch.
5. The coverage critic passes against the 88 cards (or fails loudly naming a real gap to author).
6. The vendored `engine.mjs` runs with empty `args` using Statik-generic defaults (no literal `pubmus`
   paths as the only option).
7. PUBMUS golden `engine.mjs` carries the slot markers and its gate (`--check`, `_wfcheck`, `node --test`)
   stays green.
8. `synthesis-recipe.md` (both PUBMUS and the ported extractor copy) documents the marker contract,
   the coverage critic, and the build/validation gate.
